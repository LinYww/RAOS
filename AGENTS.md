# AGENTS.md

## 1. 目标

本文件定义当前仓库顶层协作规范。AI 代理、自动化工具和参与修改的协作者在变更代码、文档、配置或 OpenSpec 文件时，都应优先遵守本规范。

核心目标只有三个：

- 保持实现正确
- 保持代码规范
- 保持改动可验证、可追溯

## 2. 工作原则

- 先理解上下文，再开始修改。
- 优先做最小必要改动，不做无关重构。
- 优先复用已有实现、已有模式和已有约定，不擅自发明新结构。
- 保持现有模块边界、目录结构、接口语义和交付流程稳定。
- 代码可读性优先于“聪明写法”。
- 未经验证，不得声称“已完成”“已修复”或“可上线”。
- 涉及架构、接口、配置、部署、数据流或任务流程的改动，必须先评估影响面。
- 改动实现时，必须同步考虑测试、文档、示例、脚本和相关说明是否需要更新。

## 3. 适用范围

本文件适用于当前仓库根目录及其子目录中的以下内容：

- 应用代码、脚本和配置文件
- 根目录协作文档与项目规范
- `openspec/` 下的 proposal、design、tasks 等变更文档
- 与开发、测试、构建、部署相关的辅助文件

## 4. 代码规范

- 保持与所在模块一致的代码风格、命名方式、目录组织和抽象层级。
- 代码规范必须符合常见企业工程实践，优先采用大多数工程师容易理解、容易维护、容易审查的写法，不引入小众、炫技或明显偏离团队常规习惯的实现风格。
- 优先选择主流、稳定、可预期的编码模式；除非现有上下文已经形成明确约定，否则避免使用过度缩写、过度链式调用、隐藏副作用写法和不必要的元编程技巧。
- 优先沿用仓库现有模式解决问题，避免只为“更优雅”而引入新的基础设施。
- 函数、类和模块职责应单一明确，避免把多个责任混在一次改动里。
- 不写无必要的兼容层、包装层或重复逻辑；已有能力可复用时必须优先复用。
- 命名必须清晰表达真实含义，禁止使用含糊缩写，除非仓库内已普遍使用。
- 新增注释应只解释“为什么”或“边界条件”，不写低价值的逐行翻译式注释。
- 注释风格、语气和粒度必须与周边代码保持一致。
- 不得为了通过检查而保留无用代码、临时分支、调试输出或注释掉的大段旧实现。
- 修改公共接口时，必须检查调用方、默认值、异常路径和兼容性影响。
- 涉及配置项时，必须明确默认行为、缺省处理和失败表现。

## 5. 文档与规范要求

- 变更接口、流程、配置、目录约定或使用方式时，必须同步更新相关文档。
- 文档应优先准确、简洁、可执行，避免空泛描述。
- 当实现行为与旧文档不一致时，应优先修正文档或明确标注差异，不能放任失真。
- 若新增规则只对局部目录生效，应放在更贴近该目录的位置，不把局部规则误写成全局规则。

## 6. 接口与兼容性约束

- 修改公共接口前，必须先确认调用方、输入输出结构、默认值、异常语义和兼容性影响。
- 非必要不得随意修改现有接口名称、参数顺序、返回结构和错误码语义。
- 如必须引入不兼容变更，必须明确说明影响范围、迁移方式和回退风险。
- 对外暴露的能力应保持行为稳定；修复问题时不得顺手改变无关行为。
- 新增接口参数或配置开关时，必须给出合理默认值，并保证旧路径在未显式启用时行为不变。

## 7. 配置与安全约束

- 新增配置项时，必须明确其用途、默认值、生效条件和错误配置下的表现。
- 配置名应清晰、稳定、可读，避免含义模糊或与现有命名体系冲突。
- 涉及环境变量、密钥、token、证书、账户或外部服务凭据时，必须使用安全占位方式，不得写入真实值。
- 不得为了调试方便降低默认安全性、关闭校验、放宽访问控制或绕过隔离逻辑。
- 涉及文件路径、网络访问、权限边界或数据读写范围时，必须显式考虑失败路径和误用风险。

## 8. 测试与验证基线

- 有现成测试体系时，优先补充或复用现有测试，不另起一套风格不同的验证方式。
- 修复 bug 时，优先增加能覆盖该 bug 的回归验证，避免只做人工口头确认。
- 新增逻辑时，应至少覆盖主路径；关键分支、边界条件和失败路径应按风险补充验证。
- 若改动影响构建、发布、部署、数据处理或任务编排，应检查上下游关键链路是否仍然成立。
- 验证结论必须基于实际执行结果；未运行的检查不能表述成“已通过”。

## 9. 明确禁止事项

除非用户明确要求，否则 AI 禁止：

- 擅自推翻现有架构、模块边界或主要流程
- 大范围重命名、批量格式化或批量重写代码风格
- 删除用户已有功能、兼容逻辑或关键上下文说明
- 绕过鉴权、隔离、访问控制、安全边界或审计要求
- 提交真实密钥、token、口令、证书或敏感配置
- 伪造测试结果、构建结果、运行结果或完成状态
- 在未说明风险、未列出未验证项的情况下宣称任务完成
- 将缓存、日志、构建产物、模型产物或无关生成文件混入提交内容
- 以“后续再清理”为由保留明显不规范的临时代码

## 10. 修改前要求

在开始实现前，AI 应优先完成以下动作：

1. 明确本次任务目标、边界和交付物。
2. 阅读与任务直接相关的代码、配置、文档和 OpenSpec 说明。
3. 找出可复用的现有实现模式、测试方式和命名方式。
4. 识别受影响的模块、接口、脚本、配置和文档。
5. 如涉及多文件联动或行为变更，先说明潜在影响面再动手。

## 11. 修改过程约束

- 每次改动都应围绕单一目标推进，避免顺手夹带额外重构。
- 如发现仓库内已有用户改动或不属于本任务的问题，默认不擅自处理，除非它直接阻塞当前任务。
- 修改时应优先保持差异清晰、范围可审查，避免一次性制造大块噪音变更。
- 引入新依赖、新脚本、新目录或新工具前，必须先确认现有能力无法满足，并说明必要性。
- 对不确定的行为假设应显式标注，不能把猜测写成事实。

## 12. 验证要求

- 改动完成后，必须执行与改动范围相匹配的验证。
- 修复 bug 时，优先验证问题是否可复现、修复后是否消失、相邻路径是否受影响。
- 新增功能时，至少验证主路径、关键边界和失败路径中的合理部分。
- 只改文档时，应检查内容与实际结构、命名和流程是否一致。
- 无法运行测试或验证时，必须明确说明原因、影响范围和残余风险。

## 13. 提交质量要求

- 最终交付内容必须能让后续协作者直接理解改动目的、范围和影响。
- 变更说明应与实际改动一致，不得使用笼统表述掩盖未完成项。
- 如存在权衡、折中或临时方案，必须明确写出原因和后续处理建议。
- 交付结果应便于审查，避免把“风格修正”和“逻辑修改”混在同一批核心变更中。

## 14. 命名规范

- 变量、函数、类、模块、配置项和文档标题名称应与其真实职责一致，避免过宽、过泛或误导性命名。
- 新增命名必须与仓库现有语言风格保持一致，不同时混用多套命名习惯。
- 非通用缩写不得直接进入公共接口、配置名或跨模块结构。
- 临时命名如 `tmp`、`misc`、`helper2`、`test123` 不得进入正式交付内容，除非该名称在局部测试上下文中确有明确语义。

## 15. 日志与错误处理规范

- 新增日志应服务于定位问题、确认状态或记录关键分支，不得为了“显得完整”而堆砌噪音日志。
- 日志内容应避免泄露密钥、token、证书、个人数据、业务敏感字段或大段原始输入。
- 错误处理应优先保证语义清晰、边界明确，不能用吞异常、静默失败或模糊兜底掩盖真实问题。
- 新增异常、错误码或失败分支时，应检查调用方是否能正确识别和处理。
- 调试输出、临时告警和一次性诊断代码在交付前必须清理干净。

## 16. 提交与说明规范

- 若任务涉及提交说明、变更摘要或 PR 描述，内容必须先写“做了什么”和“为什么做”，再写验证与风险。
- 不得使用“若干优化”“一些调整”“修复问题”等模糊表述代替真实改动说明。
- 如果改动包含限制条件、兼容性注意事项或后续待办，应在说明中明确列出。
- 若一次任务中同时包含逻辑修改和样式、格式、文案等附带调整，应在说明中区分主次。

## 17. AGENTS.md 生效与覆盖规则

- 顶层 `AGENTS.md` 默认作用于整个仓库。
- 若某个子目录存在更近一级的 `AGENTS.md`，则该子目录内的文件优先遵守更近一级规则。
- 子目录规则默认是在顶层规则基础上的补充或细化；除非明确声明，否则不应理解为完全替代顶层规则。
- 当不同层级规则发生冲突时，应优先选择更具体、影响范围更小、位置更接近目标文件的规则。
- 若发现现有规则彼此冲突且无法安全判断优先级，应暂停并与用户确认。

## 18. 完成任务前必须检查

在输出“完成”之前，AI 必须确认：

1. 改动范围与任务目标一致。
2. 代码、文档和配置变更相互一致。
3. 已执行与改动相匹配的验证。
4. 未验证项和残余风险已明确说明。
5. 没有混入无关改动、临时代码或低价值噪音。

## 19. 需要升级确认的情形

出现以下情况时，应先暂停并与用户确认，再继续推进：

- 需要引入破坏性接口变更或数据结构变更
- 需要调整架构边界、核心流程或默认行为
- 需要新增依赖、修改关键配置或改变部署方式
- 发现用户现有改动与当前任务冲突
- 无法判断哪种实现路径会对现有系统产生更小风险

## 20. 输出要求

AI 完成任务后，输出应至少包含：

- 改了什么
- 为什么这样改
- 验证了什么
- 还剩什么风险或待确认项

如任务包含规范性取舍，还应说明采用了哪些现有约定，以及为什么没有选择更大范围重构。

## 21. 与 OpenSpec 配合

当任务通过 OpenSpec 变更推进时，AI 应遵守以下附加约束：

- 先阅读 proposal、design、tasks 等上下文，再开始实现。
- 按任务清单逐项推进，避免跳过未澄清事项。
- 若实现过程中发现设计问题，应先暂停并反馈，再决定是否修改变更文档。
- 完成对应任务后，及时同步相关任务状态，避免代码与任务记录脱节。
- 当方案发生变化且确认需要更新时，同步更新 proposal、design、tasks。

## 22. 一句话原则

在当前仓库中，AI 代理的首要职责不是“尽快改完”，而是“在不破坏现有架构和交付流程的前提下，做最小、规范、可验证、可追溯的正确修改”。
